# 项目维护规则

## 信息分层

- 档案：长期稳定信息，放在 `01_profile/`。
- 日志：每天或每次事件的原始记录，放在 `02_daily/` 或专项日志。
- 知识：学习到的通用知识，放在 `07_knowledge/`。
- 决策：经过比较后决定如何做，放在 `08_questions_decisions/decisions.md`。
- 附件：已脱敏照片、视频、医院摘要、处方摘要、票据摘要，放在 `09_media/`。
- 洞察：由多个原始记录计算或归纳出的数据、报告和看板，放在 `11_insights/`；它们不是新的事实源。

## 文件角色与更新顺序

一次新信息按以下顺序维护：

1. 先把带日期的原始观察写入月度日志或所属专题记录。
2. 如果它改变了“平时是什么样”，再更新 `01_profile/baseline.md`；稳定身份变化才更新 `cat_profile.md`。
3. 如果它产生待办或明确决定，再更新 `action_queue.md` 或 `decisions.md`。
4. 最后重建派生数据、报告和 `dashboard.html`。

历史导入文件、旧 AI 回答和带日期报告保持原样追溯，不用它们覆盖更新的当前记录。无法补齐的历史信息标记为“历史缺口”，不要长期保留为开放问题。

同一数值可以在当前摘要中引用，但只在一个原始记录中维护。引用时带上日期、来源路径和数据类型：`记录值`、`观察值`、`估算值`、`推断值` 或 `参考值`。

## 命名约定

- 日期统一写 `YYYY-MM-DD`。
- 月度日志建议写成 `02_daily/logs/YYYY-MM.md`。
- 附件建议写成 `YYYY-MM-DD_简短说明.ext`。
- 不确定内容写 `待确认`，不要混入确定事实。

## 记录粒度

一次记录最好包含：

- 时间：发生日期和大致时间。
- 场景：地点、人在不在、是否有陌生人/噪音/新物品。
- 事实：小猫做了什么，持续多久，频率如何。
- 反应：你做了什么，小猫怎么回应。
- 结果：是否改善、恶化或没有变化。
- 后续：是否需要观察、调整、咨询兽医。

## 提交节奏

- 每周至少整理一次 `00_inbox/`。
- 健康、医疗、用药、疫苗、驱虫记录当天提交。
- 每次大调整后更新 `INDEX.md` 或对应目录 README。
- 更新 `11_insights/data/current_snapshot.json` 后运行 `python3 scripts/build_dashboard.py`，再用 `python3 scripts/check_repository.py` 检查生成物、链接、CSV 和公开路径。
- 本地提交可以按需进行；远端 `git push` 只有在用户明确确认后执行。

## 公开仓库检查

提交前检查：

- 是否包含照护者真实姓名、电话、邮箱、微信、住址。
- 是否包含未打码医院单据、订单号、快递单号。
- 是否包含能定位住处的照片、视频或 EXIF 信息。
- 是否把 AI 推断写成了确定事实。
